das problem
API-Keys in Chats, Notizen und Code verstreut — das ist eine Zeitbombe.
Jede Kopie eines Keys ist ein Risiko. Die Lösung: ein Ort, klare Regeln — und Verweise statt Werte.
lesson 07 · grundlagen
Eine Datei, ein Job: Zugangsdaten halten
  • Schlüssel-Wert-Paare.Eine Zeile pro Zugang: NAME=wert — mehr Format braucht es nicht.
  • Eine .env pro Projekt.Liegt im Projektordner und wird beim Start eingelesen.
  • Nie im Code, nie im Chat.Der Code liest die Variable — den echten Wert kennt nur die Datei.
▸.envmein-projekt
# Zugangsdaten dieses ProjektsOPENAI_API_KEY=sk-proj-a1b2c3…N8N_API_URL=https://dein-space.n8n.cloudN8N_API_KEY=eyJhbGciOiJIUzI1…
lesson 07 · zentrale struktur
Ein zentraler Ort — plus eine .env pro Projekt
▸claude-zentrale/ordnerstruktur
claude-zentrale/          # ein Ort — für alle Projekte  master-context.md       # zentraler Kontext, auf den CLAUDE.md verweist  zugangsdaten/    global.env            # projektübergreifende Keysprojekt-a/  .env                    # nur die Keys von Projekt Aprojekt-b/  .env                    # nur die Keys von Projekt B
Globales liegt einmal zentral — jedes Projekt behält nur seine eigenen Keys.
lesson 07 · claude.md
CLAUDE.md verweist — statt zu verraten
▸CLAUDE.mdausschnitt
## Zugangsdaten- Globale Keys: C:/claude-zentrale/zugangsdaten/global.env- Projekt-Keys: .env im jeweiligen Projektordner- Werte NIEMALS ausgeben, loggen oder committen.- Im Zweifel: Datei referenzieren, nicht zitieren.
Claude kennt den Ort — die Werte bleiben in der Datei.
lesson 07 · .gitignore
Ohne Ausnahme: .env steht in .gitignore
▸.gitignorejedes projekt
# Secrets bleiben draußen — immer.env.env.*!.env.example    # die Vorlage darf ins Repo
Einmal gepusht = verbrannt. Landet ein Key doch im Repo, hilft kein Löschen — die Git-Historie vergisst nichts. Sofort rotieren: alten Key sperren, neuen erzeugen.

lesson 07 · typische fehler
Wie Keys verbrennen — und wie nicht

✕  So nicht

  • Keys im Prompt posten — „hier ist mein Key: sk-…"
  • Im Code hardcoden — der Wert wandert in jede Kopie mit
  • .env committen — „ist doch eh mein privates Repo"

✓  Stattdessen

  • Referenzieren statt zeigen — „liegt in global.env, Zeile OPENAI_API_KEY"
  • .env.example pflegen — jeder neue Schlüssel auch in die Vorlage
  • Keys rotieren — regelmäßig, und sofort bei jedem Verdacht
