lesson 06 · die vier begriffe
Vier Begriffe reichen für den Start
Der Projektspeicher

Repository

Dein Projekt samt kompletter Historie — lokal bei dir und als Kopie auf GitHub.

Der Sicherungspunkt

Commit

Ein benannter Zwischenstand („Login gebaut") — zu jedem kannst du zurückspringen.

Das Hochladen

Push

Schiebt deine lokalen Commits zu GitHub — erst dann ist der Stand auch online gesichert.

Die Ausnahme-Liste

.gitignore

Legt fest, was Git ignoriert: Pakete, Build-Müll — und vor allem Zugangsdaten.

lesson 06 · demo
Vier Befehle — und alles ist gesichert
terminal — mein-projekt
❯ git init
✓ Repository angelegt — ab jetzt wird Historie geschrieben
❯ git add .
❯ git commit -m "Erster Stand"
✓ 14 Dateien im ersten Commit gesichert
❯ git push -u origin main
✓ Stand liegt auf GitHub — Sicherungsnetz aktiv
Merken musst du dir das nicht: Claude Code führt diese Befehle auf Zuruf für dich aus.

lesson 06 · claude übernimmt
Git-Arbeit? Delegierst du.
  • Committen lassen.„Committe den aktuellen Stand" — Claude führt die Git-Befehle für dich aus.
  • Commit-Messages schreiben lassen.Claude kennt die Änderung und fasst sie präziser zusammen als „Update".
  • Vor großen Änderungen committen.Erst Sicherungspunkt setzen, dann umbauen lassen — der Rückweg bleibt offen.
  • Du gibst frei.Claude fragt vor jedem Git-Kommando nach — du behältst die Kontrolle.
lesson 06 · typische fehler
Woran Git-Anfänge scheitern

✕  So nicht

  • Riesen-Sammel-Commits — „alles von dieser Woche" in einem Rutsch
  • Secrets im Repo — Keys und Passwörter mitcommitten
  • „Später mal sichern" — und der gute Stand ist längst überschrieben

✓  Stattdessen

  • Kleine Commits — ein Arbeitsschritt, ein Sicherungspunkt
  • Sprechende Messages — „Login-Formular validiert" statt „Update"
  • Vor jedem Experiment committen — dann darf es auch schiefgehen
