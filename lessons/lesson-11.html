lesson 11 · artefakte
Was ist ein Artefakt?
Ein sichtbares, prüfbares Zwischenergebnis — deine Entscheidungsgrundlage, bevor gebaut wird.
zum ansehen

Design-Preview

Sehen, wie es aussieht — als HTML-Vorschau, bevor eine Zeile echter Code entsteht.

zum verstehen

Technisches Konzept

Architektur & Entscheidungen schwarz auf weiß — diskutierbar, solange nichts festbetoniert ist.

zum steuern

Umsetzungsplan

Schritte, Reihenfolge, Risiken — als PLAN.md direkt im Projekt.

lesson 11 · phasenplan
So sieht der Phasenplan konkret aus
▸PLAN.mdmein-projekt
## Phase 1 — UI-KonzeptArtefakt: Design-Preview (HTML)Status:   freigegeben ✓ ## Phase 2 — DatenmodellArtefakt: Technisches KonzeptStatus:   in Prüfung … # Phase 3 & 4: gesperrt bis Freigabe von Phase 2
Jede Phase hat ein Artefakt und einen Status — der Plan lebt als PLAN.md im Projekt.
lesson 11 · freigabe-gate
Warum das Gate zwischen den Phasen?

✕  Ohne Gate

  • Claude baut alles am Stück durch
  • Fehler wandern mit — von Phase zu Phase
  • Am Ende: der große Umbau

✓  Mit Gate

  • Jede Phase wird abgenommen — von dir
  • Fehler sterben früh — solange sie billig sind
  • Du behältst die Richtung in der Hand
lesson 11 · demo
So klingt der Prompt
claude code — mein-projekt
❯ „Plane das Projekt in Phasen. Erstelle zuerst
  NUR ein Design-Artefakt als HTML-Preview.
  Baue noch nichts."

✓ PLAN.md erstellt — 4 Phasen, je ein Artefakt
✓ Artefakt folgt: design-preview.html
Die wichtigste Zeile steht am Ende: „Baue noch nichts." — das Artefakt kommt zuerst.
aus der praxis
Genau so ist übrigens diese Schulung entstanden: erst Design-Preview, dann Freigabe, dann Umsetzung.
Das Verfahren skaliert — vom Ein-Tages-Feature bis zum Monatsprojekt.
lesson 11 · recap
Das nimmst du mit
  • Erst Phasen planen — nicht sofort bauen lassen
  • Je Phase ein Artefakt — sichtbar, prüfbar, diskutierbar
  • Freigabe-Gate: prüfen, Feedback, dann erst weiter
  • Fehler sterben früh — nicht erst beim großen Umbau
